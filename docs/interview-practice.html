What this system does

It simulates an AI-native integration hub: connect to multiple external partner APIs, let an agent automate sync and publish workflows, ground answers in company docs, and require human approval before side effects (like uploading a creative).

The fake partners — PulseAds (Bearer + cursor pagination + 429 retries), NovaReach (API-key + offset pagination), and CreativeBox (basic auth + multipart upload) — are deliberately different so the connector abstraction has to earn its keep.

Goal → Agent → Skill → Connector → Partner API

Side-effecting steps (e.g. publish creative) pause at a human-in-the-loop gate until you Approve or Deny in the dashboard or API. Read-only steps (sync, search docs) run immediately. Everything is logged in an inspectable run trace.

By default: FakeLLM (deterministic planner), HashEmbedder (offline embeddings), and mock_apis/ (local FastAPI fakes). No keys, no cloud, reproducible in CI and in an interview room.

How to run it

From PowerShell (Windows) or any shell — repo root: ai-integration-sandbox/

One-time setup

python tasks.py setup

Full stack (3 terminals)

# Terminal A — fake partner APIs (:9000)
python tasks.py mock-apis

# Terminal B — FastAPI hub (:8000)
python tasks.py run

# Terminal C — React dashboard (:5173)
python tasks.py ui

Open http://localhost:5173 — UI proxies API calls to the service.

Verify & rehearse offline

python tasks.py test          # full suite, no network
python tasks.py eval          # RAG + agent + generation scorecard
python tasks.py drill-katas   # 7 failing katas to solve
python tasks.py deploy-local  # LocalStack smoke (skips if Docker down)

API docs when the service is running: http://127.0.0.1:8000/docs

How I use this to practice interviews

The repo is not just code to read — it's a rehearsal instrument. I alternate between running the system, solving drills, and narrating architecture as if an interviewer is watching.

Weekly rhythm (suggested)

Day 1 — Run & narrate (45 min) Boot mock-apis + service + UI. Start a publish run, approve it, point at trace steps. Say out loud: plan → skill → gate → connector → partner. Record yourself or use a timer.
Day 2 — Factorial depth (45 min) Open specs/agent.md and specs/rag.md. Run python tasks.py eval. Explain hybrid vs pure BM25/dense using docs/rag-notes.md. Demo HITL deny path (no side effects).
Day 3 — Oxigent depth (45 min) Read connector transport (retries, 429, circuit breaker). Walk tests/connectors/. Show dashboard metrics. Sketch deploy from docs/architecture.md.
Day 4 — System design (30 min) Pick one prompt from drills/system-design/. Whiteboard for 20 min, then open the <details> model answer and compare.
Day 5 — Katas (60 min) python tasks.py drill-katas — solve one or two failing tests (backoff, RRF, pagination…). Check SOLUTION_*.md only after attempting.
Day 6 — Challenge-back (20 min) Two scenarios from drills/challenge-back/. State stakeholder claim, counter-argument, alternative.
Day 7 — Mock interview (60 min) Full demo script below + 10 flashcards from drills/flashcards.md. No notes for the first 15 min.

Before each real interview

  • Run python tasks.py test — green suite = confidence to run live
  • Pick 3 modules to foreground based on which company is first
  • Prepare one “I extended this” story (e.g. added eval threshold, fixed a connector, solved a kata)
  • Know where specs live — Factorial cares that you spec before you code

Live demo script (~5 minutes)

Use this when an interviewer says “walk me through something you built” or “show it running”.

  1. Context (30 s) — “This is an offline integration hub: three fake ad networks, one agent, human approval on writes, hybrid RAG over partner docs.”
  2. Dashboard metrics (30 s) — Open UI. Point at total runs, success rate, estimated value. “This is the monitoring layer Oxigent-style integrations need.”
  3. Read path (60 s) — Start run: sync campaign data from pulseads. Show trace: plan → skill → connector. No approval needed.
  4. RAG (45 s) — POST /search or ask docs in UI if wired — show citations and mention BM25 + dense + RRF from rag/.
  5. Write path + HITL (90 s) — Start: publish the new creative to creativebox. Show pending approval. Approve → creative lands on mock store. Mention deny path leaves zero side effects.
  6. Framework (45 s) — GET /connectors and GET /skills. “Adding a partner is a module + registry entry; skills mirror that for agent capabilities.”
  7. Quality (30 s) — Mention eval harness, committed thresholds, spec-first in specs/.
  8. Close (30 s) — “Default is FakeLLM and mock APIs so this runs in CI and in this room without keys; real providers are behind env flags.”

Talking points cheat sheet

  • Why offline-first? Deterministic tests, no interview-day network failures, cheap CI evals.
  • Why hybrid RAG? BM25 for IDs and rare tokens; dense for paraphrase; fusion when either alone fails.
  • Why HITL? Side effects are low-reversibility; model can hallucinate tool args.
  • Why MCP? Same integrations exposed to IDE agents and external LLM clients with typed tools.
  • Why specs first? Aligns team on success criteria before code; specs are the contract for evals too.
  • Why SQS for approvals? Durable async queue; see ADR 001. In-process gate for MVP.
  • Why S3 for creatives? Blobs not rows; see ADR 002.
  • Connector resilience? Backoff + jitter, Retry-After on 429, circuit breaker per partner.
