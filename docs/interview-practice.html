    What this system does

    It simulates an AI-native integration hub: connect to multiple external partner APIs, let an agent automate sync and publish workflows, ground answers in company docs, and require human approval before side effects (like uploading a creative).

    The fake partners — PulseAds (Bearer + cursor pagination + 429 retries), NovaReach (API-key + offset pagination), and CreativeBox (basic auth + multipart upload) — are deliberately different so the connector abstraction has to earn its keep.

    Side-effecting steps (e.g. publish creative) pause at a human-in-the-loop gate until you Approve or Deny in the dashboard or API. Read-only steps (sync, search docs) run immediately. Everything is logged in an inspectable run trace.

    By default: FakeLLM (deterministic planner), HashEmbedder (offline embeddings), and mock_apis/ (local FastAPI fakes). No keys, no cloud, reproducible in CI and in an interview room.

    Interview Q&A — 15 questions, crisp answers

    Click a question to expand the answer. Cover these before every interview — they map directly to the diagrams, specs, and code.

    System overview

    • Walk me through what this system does end-to-end.

      A goal string arrives at POST /runs. The agent assembles working memory (recent episodes, relevant docs, skill procedures) then asks the LLM to select a skill via function-calling. Read-only skills (sync, search) execute immediately through the connector framework — retry, backoff, circuit-breaker included. Side-effecting skills (publish_creative) pause at the HITL gate: the caller must POST /runs/{id}/approve before any connector is touched. Every step is appended to an immutable run trace in the SQLite ledger, which feeds episodic memory for future runs.

    • Why offline-first? Why FakeLLM and mock APIs?

      Deterministic tests that run anywhere without API keys or network — in CI, in a local dev environment, and in an interview room. FakeLLM returns predictable skill selections by keyword matching; HashEmbedder is a deterministic hashed bag-of-words. Swapping in real providers is an env-flag change (AIH_LLM_PROVIDER=anthropic), not a code change. The principle: domain logic should not be coupled to provider availability.

    • What is Spec-Driven Development in this repo?

      No code is written before its spec exists in specs/. Each spec defines: Goal, Inputs/Outputs, Behaviour (numbered steps), Constraints, Failure modes, Success criteria (measurable), and Out of scope. The Success criteria become the eval thresholds committed in evals/thresholds.json. This aligns on "done" before implementation starts and makes specs the living contract for both code and evals.

    Agent & HITL

    • What is the HITL gate? When does it trigger?

      Every Skill declares side_effect: bool. When the agent selects a side-effecting skill, it constructs an ApprovalRequest (what / why / payload preview / reversibility) and passes it to the injected Approver. The loop pauses until Approver.decide() returns. Only if decision.approved is True does the skill run. A denial records the decision to the trace with zero side effects — no connector is called. Approver implementations: AutoApprover (tests), CLIApprover (stdin), APIApprover (awaits POST /runs/{id}/approve).

    • How does the agent select tools?

      The planner builds ToolSpecs from each skill's Pydantic input schema (field name, type, description) plus a finish tool. It assembles a prompt with these specs and calls LLMClient.tool_call(prompt, tool_specs) which returns a (tool_name, args) pair. FakeLLM matches the goal string to a skill by keyword; a real LLM uses function-calling. The loop continues until the same tool+args is re-selected (convergence) or finish is selected.

    • What happens if the agent exceeds its step budget?

      The orchestrator increments a step counter each iteration. If it reaches agent_max_steps, the loop terminates with status="max_steps" and returns the partial RunResult with the full trace so far. The trace is still persisted to the ledger. This bounds runaway loops — critical when a real LLM makes unexpected or looping tool selections.

    Hybrid RAG

    • Why hybrid RAG? Why not just BM25 or just dense embeddings?

      BM25 excels at exact-token matches (error codes, IDs, rare terms like "429") but has no semantics — "rate limit" and "429" are unrelated to it. Dense embeddings capture paraphrase and synonyms but miss rare tokens and depend on model quality. The offline HashEmbedder is a deterministic hashed bag-of-words with no IDF — deliberately making BM25 and dense disagree so the tests can prove fusion beats either alone by MRR. In production, the two signals complement each other and fusion robustly improves recall.

    • What is RRF vs alpha fusion? When do you use each?

      Alpha fusion: fused = α × dense_norm + (1-α) × sparse_norm over min-max-normalized scores. Blends magnitudes — needs calibration and is sensitive to score distributions. RRF: score = Σ 1/(k + rank_i) over each retriever's ranking. Blends ranks, not magnitudes — scale-free and robust when BM25 and cosine scores live on incomparable scales (the normal case). Rule of thumb: use RRF by default; use alpha when both signals are well-calibrated and you want explicit lexical/semantic balance. At the extremes, alpha=0 is pure BM25, alpha=1 is pure dense.

    • What is the deterministic path in RAG?

      If the query contains a structured entity reference (a campaign ID), the HybridRetriever calls an injected resolver that fetches the authoritative record directly from the connector. This record is returned alongside retrieved text chunks, labelled connector:<partner> for provenance rather than doc:<id>. The resolver is injected (not called directly) so RAG stays decoupled from the connector/network layer. Rule of thumb: retrieve prose, look up facts.

    Connectors & resilience

    • How does the connector framework handle retries and rate limiting?

      Every connector uses a shared Transport wrapping httpx.AsyncClient. On HTTP 429, Transport reads the Retry-After header and sleeps for that duration (sleep is injectable so tests run instantly). On 5xx or network errors, it applies exponential backoff with jitter up to a configured budget. 4xx errors (401, 404) are not retried — they surface as typed errors (AuthError, NotFoundError). After N consecutive failures, a circuit breaker opens and fails fast with CircuitOpenError until a cooldown period passes.

    • How do you add a new (4th) partner connector?

      (1) Create src/aih/connectors/newpartner.py — a Connector subclass with a ConnectorConfig and the partner's auth strategy. (2) Implement get_records() using cursor_paginate or offset_paginate from paginate.py. (3) Register: registry.register("newpartner", NewPartnerConnector). (4) Add credentials to config.py and .env.example. (5) Add a mock route to mock_apis/app.py. (6) Add a corpus doc if RAG coverage is needed. The runbook at docs/runbooks/add-a-partner.md walks this step-by-step.

    • What is idempotency in the connector context?

      Before a creative is pushed, an Idempotency-Key header is computed as a hash of the payload content. If the same creative is uploaded twice (e.g. after a network-timeout retry), the server deduplicates based on this key and returns the original result without storing a duplicate. This guarantees at-most-once side effects even under retry logic — important when retrying a write that may have partially succeeded on the first attempt.

    Evals, Skills & Specs

    • What eval suites does the harness run?

      Five suites: (1) retrieval — recall@k, MRR, nDCG@k against labelled doc IDs. (2) generation — answer_from_docs answers scored by an LLM-as-judge using rubric keyword coverage. (3) tool_selection — agent planner accuracy on expected skill per goal. (4) red-team — guardrail block rate on injection/jailbreak prompts. (5) online sampling — sample live runs to a HITL review queue. Thresholds are committed in evals/thresholds.json; the runner exits non-zero if any metric falls below threshold.

    • How do you prevent eval regression across model or retriever changes?

      Thresholds are committed to evals/thresholds.json alongside the code. The eval runner (python tasks.py eval) exits with code 1 if any metric falls below its threshold — CI enforces this. When a change improves scores, thresholds are bumped in the same commit — scores can only go up. HITL review exports (CSV + JSON) include a blank human_score column; reviewed files are re-ingested to merge human judgements back into the dataset.

    • What is an AI Skill and how does it differ from an MCP tool?

      A Skill is a first-class domain capability with a typed Pydantic input_model and output_model, a side_effect flag, and a run(payload, ctx) entry point. Skills live in the SkillRegistry and are invoked by the agent orchestrator loop — with HITL gating enforced for side_effect=True. An MCP tool is a protocol-level binding that exposes the same underlying logic over the MCP stdio protocol to external LLM clients (IDE agents, Claude Desktop). Same logic, different protocol surface — one mechanism, multiple faces.

    Talking points cheat sheet

    • Why offline-first? Deterministic tests, no interview-day network failures, cheap CI evals. Real providers are behind env flags — one change to swap in.
    • Why hybrid RAG? BM25 wins on exact token matches (IDs, error codes, rare terms); dense wins on paraphrase and semantic queries. Fusion beats either alone in MRR — proven by the test suite.
    • Why RRF over alpha fusion? RRF blends ranks, not magnitudes — scale-free and robust when BM25 and cosine scores live on incomparable scales. Alpha needs calibrated, comparable distributions.
    • Why HITL? Side effects are low-reversibility; models can hallucinate tool arguments. The gate creates an auditable paper trail: every decision is recorded to the run trace.
    • Why MCP? Same integrations exposed to IDE agents and external LLM clients with typed tool envelopes — no additional HTTP surface needed.
    • Why specs first? Aligns the team on success criteria before code; the Success criteria become the eval thresholds. Specs are a living contract, not a one-time artefact.
    • Why SQS for approvals in prod? Durable async queue survives service restarts; approval decisions arrive out-of-band. In-process APIApprover is the MVP — ADR 001 explains the upgrade path.
    • Why S3 for creatives? Blobs not rows: unbounded size, cheap storage, CDN-friendly. ADR 002 explains why DynamoDB wasn't the right fit here.
    • Connector resilience? Exponential backoff + jitter, Retry-After on 429, circuit breaker per partner (opens after N failures, fails fast until cooldown). Sleep is injectable so tests run instantly.
    • Why idempotency keys on writes? Deduplicates retried writes server-side: the connector hashes payload content → Idempotency-Key header. Guarantees at-most-once side effects under retry logic.
    • Deterministic vs probabilistic retrieval? Text retrieval returns the likely relevant passage. For structured facts (a campaign ID, a price, a status), look the record up in the authoritative connector — return it with connector: provenance, not a retrieved paragraph.
    • How do evals guard against regression? Thresholds committed to evals/thresholds.json; CI exits non-zero on any metric below threshold. Scores can only go up — threshold bumps in the same commit as improvements.
    • What is an AI Skill vs an MCP tool? A Skill is a first-class domain capability with typed I/O, a side_effect flag, and HITL enforcement. An MCP tool exposes the same underlying logic over the MCP stdio protocol to external LLM clients.
    • Why FastAPI for the AI layer? Native async matches async connectors, RAG, and agent loops. SSE pushes run state without polling. httpx.ASGITransport tests the full app in-process — no sockets in CI.
